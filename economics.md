# Economics

> Transparent referral mechanics, settled on-chain. Every split is disclosed before payment, recorded after.

---

## Referral Mechanics

```
Agent discovers service (via Scout, registry, or direct)
→ Agent pays via x402 ($0.10 example)
→ On-chain settlement on Base (~2 seconds)
→ Smart contract splits:
    85% → service provider wallet
    15% → referring wallet (or 0% if no referrer)
→ Split is disclosed in:
    - 402 response headers (before payment)
    - contract events (after payment)
    - Igniter README
    - Verifier metadata
```

---

## Default Parameters

| Parameter | Default | Range | Notes |
|-----------|---------|-------|-------|
| Referral split | 15% | 0–100% | Configurable by service provider |
| Verifier-recognized max | 50% | — | Above 50% flagged as promotional |
| Promotional splits | Allowed | — | Must be disclosed in 402 headers with end date |

---

## Split Semantics

Three cases, treated differently:

| Recipient | When | Disclosed As |
|-----------|------|-------------|
| Referring agent wallet | Agent A routes to service via Scout | `referral_agent` |
| Infrastructure wallet (Scout) | Scout receives cut for routing | `referral_infrastructure` |
| No referral | Agent discovers service directly | `referral_split: 0` |

## 402 Response Disclosure

Every x402 payment response generated by Kindling Igniter includes:

```json
{
  "maxAmountRequired": "100000",
  "asset": "USDC",
  "payTo": "0x...",
  "chainId": 8453,
  "referral_split_pct": 15,
  "referral_wallet": "0x...",
  "referral_disclosure": "15% of payment settles to the referring wallet. Built with Kindling Igniter. Economics are on-chain and auditable. See: github.com/kind-ling/igniter",
  "built_with": "@kindling/igniter@0.1.0"
}
```

This disclosure appears **before payment** — agents know the split before committing.

## Verifier Treatment of High Splits

| Split | Verifier Treatment |
|-------|-------------------|
| 0–50% | Included normally in demand score |
| >50% | Flagged as `promotional_split: true`, excluded from demand score normalization |
| Temporary promotional (disclosed) | Allowed, flagged with end date |

High splits are not penalized — they're labeled. The Verifier scores real demand from real unique payers, not wash volume inflated by referral incentives.

---

## Self-Traffic Policy

Services must disclose wallets they control. Payments between affiliated wallets:

- Labeled `self_traffic: true` in Verifier data
- **Excluded** from unique-payer counts (the primary anti-sybil measure)
- Displayed separately — volume is visible, but not counted as demand from independent payers

---

## Routing Policy (Phase 3 — Scout)

When Scout routes a task, selection follows this strict order:

1. **Eligibility filter** — service must support the requested task type
2. **Quality floor** — Verifier scores must meet agent-specified or system-default minimums
3. **Agent preference optimization** — route according to agent's stated weights (e.g., "prioritize outcome over price")
4. **Economic consideration** — price and availability, only after quality and fit thresholds are met
5. **First-party tie-break disadvantage** — equivalent services: non-first-party wins

Referral split size has **zero effect** on routing position. This is enforced in code, not policy alone.

---

## On-Chain Verification

All referral splits are settled via smart contract on Base mainnet. Contract address published before Phase 1 launch. Events are public and auditable by anyone.
